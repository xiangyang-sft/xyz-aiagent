## Step 2：核心概念深入

### 2.1 单 Agent 应用的架构分层

一个工程级的单 Agent 应用应该分为四层：

```
┌──────────────────────────────────────┐
│           层 1：用户接口层             │
│  CLI / API / Web / GUI               │
│  负责接收用户请求、展示结果             │
├──────────────────────────────────────┤
│           层 2：Agent 核心层           │
│  ReAct 循环 + 推理 + 规划            │
│  这是 Agent 的"大脑"                  │
├──────────────────────────────────────┤
│           层 3：工具层                 │
│  工具注册 / 工具执行 / 错误处理        │
│  这是 Agent 的"手和脚"                │
├──────────────────────────────────────┤
│           层 4：记忆层                 │
│  短期记忆 / 长期持久化 / RAG 检索     │
│  这是 Agent 的"记忆"                  │
└──────────────────────────────────────┘
```

### 2.2 关键设计决策

在设计一个 Agent 应用时，需要回答以下问题：

| 问题 | 选项 | 取舍 |
|------|------|------|
| **LLM 调用方式** | 同步 / 流式（SSE） | 同步简单，流式用户体验好 |
| **工具注册方式** | 硬编码 / 动态注册 / MCP | 硬编码最简单，MCP 更灵活 |
| **记忆持久化** | JSON 文件 / SQLite / 向量库 | JSON 最简单，SQLite 适合结构化，向量库适合 RAG |
| **状态管理** | 函数式（每次传入） / 类封装 / 全局变量 | 类封装最常用 |
| **错误处理** | 抛出异常 / 返回错误消息 / 重试机制 | 多策略组合 |
| **循环控制** | 固定轮次 / 动态停止 / Token 预算 | 动态停止 + Token 预算最佳 |
| **配置管理** | 硬编码 / 环境变量 / 配置文件 / 数据库 | 环境变量 + 配置文件 |

### 2.3 ReAct 循环的工程化实现

之前我们实现的 ReAct 循环是"白板实现"，现在需要加工程化改进：

```
之前（教学版）：
  while True:
      response = llm(messages + tools)
      if no_tool_call → break

现在（工程版）：
  while True:
      # 1. Token 预算检查
      if total_tokens > MAX_TOKENS → 强制结束
      
      # 2. 注入记忆
      context = short_term.get() + long_term.query(user_id) 
      
      # 3. LLM 推理
      response = llm(messages + context + tools)
      
      # 4. 处理工具调用
      if tool_call:
          result = execute_tool(tool_call)
          observation = process_error(result)  # 错误处理
          long_term.remember(user_id, observation)  # 记忆更新
          
      # 5. 死循环检测
      if repeated_pattern_detected() → 强制退出
      
      # 6. 轮次限制
      if step > MAX_STEPS → 强制结束
```

### 2.4 工具设计的工程原则

一个好的工具设计应该遵循：

| 原则 | 说明 | 反例 | 正例 |
|------|------|------|------|
| **单一职责** | 一个工具做一件事 | `process_data(type, data)` 参数决定做不同的事 | `calculate(expr)` 只计算，`search(keyword)` 只搜索 |
| **明确输入** | 参数清晰、类型明确 | `data: str` 任意字符串 | `expression: str` 数学表达式 |
| **可预期的输出** | 返回格式固定 | 不同情况返回不同结构 | 始终返回 `{"success": bool, "result": any}` |
| **错误友好** | 失败时返回有用信息 | 抛出异常 | 返回 `{"success": false, "error": "..."}` |
| **幂等性** | 同输入同输出（读操作） | 写操作也有副作用 | 读操作幂等，写操作明确标注 |

### 2.5 安全与限制

生产级 Agent 必须考虑以下安全措施：

```
┌────────────────────────────────────┐
│  🛡️ Agent 安全防护                  │
│                                    │
│  1. 工具权限控制                    │
│     • 读/写分离：某些工具只读        │
│     • 路径白名单：只允许操作指定目录  │
│                                    │
│  2. 执行限制                        │
│     • 最大工具调用次数：防止无限循环  │
│     • 最大 Token 消耗：控制成本      │
│     • 工具调用超时：防止卡死          │
│                                    │
│  3. 内容过滤                        │
│     • 敏感操作确认：删除/覆盖前询问   │
│     • 输出长度限制：防止刷屏          │
│                                    │
│  4. 降级策略                        │
│     • 工具失败时 Agent 应能继续      │
│     • 出错时返回友好提示              │
└────────────────────────────────────┘
```

## 🎯 面试题

### Q1：你在实际构建一个单 Agent 应用时，最需要考虑的工程问题有哪些？

**参考答案：** 除了功能实现外，还需要考虑：
1. **健壮性**：工具调用失败时的降级策略
2. **成本控制**：Token 预算管理和循环轮次限制
3. **死循环防护**：模式检测和强制退出机制
4. **记忆持久化**：跨会话保持上下文
5. **配置管理**：模型选择、工具开关等可配置
6. **可观测性**：Log 记录每个步骤的输入输出，便于调试
7. **安全约束**：文件路径白名单、敏感操作确认

### Q2：Agent 应用中的"工具错误处理"应该怎么做？工具返回错误后 Agent 应该怎么办？

**参考答案：** 分层处理策略：
- **1. 重试**：网络类错误自动重试 1-2 次
- **2. 降级**：工具不可用时，Agent 告知用户并提供替代方案
- **3. 上下文保持**：出错信息被注入下一轮 LLM 调用，让 LLM 自行决定如何应对
- **4. 日志记录**：记录错误类型、工具名称、参数和堆栈
- 关键原则：**不要让工具错误直接传递到用户端**，Agent 应当消化并友好告知

### Q3：你如何决定一个 Agent 应用的工具粒度？一个工具应该多小？

**参考答案：** 遵循 **单一职责原则**：
- 一个工具对应一个不可再拆的原子操作
- 如果工具描述中出现"和"（and），说明应该拆成两个工具
- 判断标准：如果 LLM 不需要在一个工具调用里完成两件独立的事，就拆
- 实际做法：先粗粒度实现，使用过程中发现 LLM 经常"同时调用两个相关工具"，就考虑合并

### Q4：Agent 的短期记忆和长期记忆如何配合工作？具体如何实现？

**参考答案：**
- **短期记忆**：当前对话的 Message 列表（sliding window），每次循环都传入
- **长期记忆**：SQLite/JSON 文件，在每次对话开始前读取，对话结束后写入
- **配合流程**：
  1. 对话开始时，加载长期记忆中与用户相关的信息
  2. 注入到 System Prompt 中作为上下文
  3. 对话过程中，短期记忆维护对话历史
  4. 对话结束时，提取关键信息写入长期记忆
- **关键点**：长期记忆读取发生在 LLM 调用之前（减少 Token），写入发生在对话结束之后

### Q5：如何为 Agent 设计一个好的 System Prompt？有什么技巧？

**参考答案：**
1. **身份定义**：明确你是谁、能做什么、不能做什么
2. **行为规范**：回复风格、态度、限制条件
3. **工具说明**：何时使用什么工具，工具优先级
4. **处理流程**：思考 → 行动 → 观察 → 回答的循环说明
5. **边界约束**：最大调用次数、安全限制、隐私保护
6. **示例引导**：1-2 个典型交互示例

**技巧：**
- System Prompt 中最重要的信息放最前面（LLM 更关注开头和结尾）
- 用分隔符（---、```）清晰划分不同段落
- 避免否定句式（"不要做 X" → 改为 "应该做 Y"）
- 定期评估 System Prompt 效果，迭代优化
