## Step 1：基础知识 — 为什么需要多 Agent？

### 1.1 单 Agent 的天花板

单个 Agent 虽然强大，但在复杂场景中会遇到明显的瓶颈：

| 问题 | 表现 | 类比 |
|------|------|------|
| **认知带宽有限** | 一个 LLM 实例上下文窗口有限，同时处理多项任务时注意力分散 | 一个人同时写三份报告 |
| **专业能力冲突** | 同一个 LLM Prompt 既要做代码评审又要做架构设计，提示相互干扰 | 让全科医生做心脏手术 |
| **单点故障** | Agent 出错时无人校验，错误被直接传递给用户 | 没有 Code Review 的代码提交 |
| **串行效率低** | 即使任务可并行，单 Agent 也得一步步做 | 流水线上只有一个工人 |

**一句话：一个天才不如一个团队。**

### 1.2 多 Agent 协作的核心思想

**多 Agent 系统（MAS, Multi-Agent System）** 的核心思想：

> 将复杂任务分解，交给多个专业 Agent 协作完成，通过角色分工、消息通信、协调机制实现 1+1 > 2 的效果。

```
传统单 Agent：
  ┌───────────────┐
  │  🧠 一个 Agent │ ← 什么都会，什么都不精
  │  写代码+测试   │
  │  +部署+监控    │
  └───────────────┘

多 Agent 协作：
  ┌──────────┐  ┌──────────┐  ┌──────────┐
  │ 👨‍💻 编程Agent│  │ 🧪 测试Agent│  │ 🚀 部署Agent│
  │  专业写代码  │  │  专注测试  │  │  负责部署  │
  └─────┬─────┘  └─────┬─────┘  └─────┬─────┘
        │ 代码PR        │ 测试报告      │ 部署状态
        └───────┬───────┴───────┬───────┘
                │    🧠 协调者   │
                │ 分配任务/汇总结果 │
                └───────────────┘
```

### 1.3 多 Agent 系统的分类

按**协作关系**可分为三大类：

| 类型 | 描述 | 类比 | 代表框架 |
|------|------|------|----------|
| **🧭 编排式（Orchestration）** | 一个中心 Controller 分配任务、调度 Agent、汇总结果 | 项目经理 + 开发团队 | AutoGen, CrewAI, LangGraph |
| **🤝 协商式（Debate/Collaboration）** | 多个 Agent 平等对话，通过讨论/辩论达成共识 | 圆桌会议 | ChatDev, Multi-Agent Debate |
| **🏛️ 层级式（Hierarchical）** | 多级管理者逐层分解任务，形成树状结构 | 公司组织架构 | MetaGPT, BabyAGI |

按**通信方式**分类：

| 方式 | 特点 | 适用场景 |
|------|------|----------|
| **共享上下文（Shared Context）** | 所有 Agent 读写同一份上下文/黑板 | 协作式编辑、信息汇总 |
| **消息传递（Message Passing）** | Agent 之间通过消息队列通信 | 分布式系统、流水线任务 |
| **函数调用（Tool Call）** | Agent 通过调用其他 Agent 暴露的工具来协作 | 嵌套 Agent、插件架构 |

### 1.4 核心挑战

| 挑战 | 描述 | 解决思路 |
|------|------|----------|
| **任务分解** | 如何把大任务合理拆分成子任务 | 结构化 Prompt、Chain of Thought |
| **结果一致性** | 不同 Agent 给出相互矛盾的结论 | 投票机制、仲裁规则 |
| **通信开销** | Agent 之间交流过多，消耗大量 Token | 消息精简、结构化传输 |
| **死锁/循环** | Agent 互相等待或陷入无限讨论 | 超时机制、最大轮次限制 |
| **角色冲突** | Agent 角色定义不清导致功能重叠 | 明确的角色描述和职责边界 |

## Step 2：核心概念深入

### 2.1 三种主流协作模式详解

#### 🧭 编排式（Orchestration）— 最常用

**核心流程：**

```
用户 → Controller (LLM) ─┬→ Agent A (任务1) ─┐
                         ├→ Agent B (任务2) ─┤
                         ├→ Agent C (任务3) ─┤
                         └→ Agent D (任务4) ─┘
                              ↓ 结果汇总
                         Controller → 最终输出
```

**工作流程：**
1. **接收任务**：Controller 分析用户请求
2. **规划分解**：将任务拆成子任务，确定依赖关系
3. **分配执行**：将子任务分发给对应的 Specialist Agent
4. **结果汇聚**：收集各 Agent 结果，进行整合汇总
5. **验证反馈**：检查结果完整性，必要时要求重做

**优缺点：**
- ✅ 结构清晰，易于理解和调试
- ✅ 适合有明确流程的任务（代码开发、报告生成）
- ❌ Controller 可能成为瓶颈
- ❌ 灵活性较低，难以处理动态变化

#### 🤝 协商式（Debate/Collaboration）— 最接近人类团队

**核心流程：**

```
         ┌───────────── 讨论 / 辩论 ─────────────┐
         │                                       │
    Agent A ─────────────────────────────────── Agent B
    (不同观点)                                  (不同观点)
         │                                       │
         └────────── 仲裁/投票 → 统一答案 ──────────┘
```

**几种模式：**

| 子模式 | 描述 | 示例 |
|--------|------|------|
| **多角色讨论** | 不同角色从不同角度分析问题 | 产品经理 + 开发 + 测试讨论需求 |
| **辩论式** | 多个 Agent 轮流反驳对方观点直到共识 | 正反方辩论一个方案 |
| **投票式** | 多个 Agent 独立给出答案 + 投票选最优 | 多个 Analyst 预测股票 |

**优缺点：**
- ✅ 结果更可靠（多人校验）
- ✅ 能发现单 Agent 忽略的问题
- ❌ Token 消耗巨大（反复讨论）
- ❌ 可能陷入无休止辩论

#### 🏛️ 层级式（Hierarchical）— 最适合大型项目

**核心流程：**

```
        ┌────────── CEO Agent ──────────┐
        │        (战略规划)               │
        ↓                                ↓
   ┌─── CTO Agent ───┐          ┌─── CFO Agent ───┐
   │  (技术分解)      │          │  (资源规划)      │
   ↓       ↓         ↓          ↓                 ↓
   Dev    QA    DevOps        Budget   Cost        ...
   Agent  Agent  Agent         Agent   Agent
```

**优缺点：**
- ✅ 天然适合大型复杂项目
- ✅ 可扩展性好（只需增加层级/节点）
- ❌ 实现复杂度高
- ❌ 层级越多，延迟越大

### 2.2 通信与消息协议

#### 结构化消息格式

Agent 之间不应使用自然语言随心所欲地聊天，而是用结构化格式：

```json
{
  "message_id": "msg_001",
  "sender": "architect_agent",
  "receiver": "developer_agent",
  "msg_type": "task_assignment",
  "content": {
    "task_id": "T-003",
    "description": "实现用户登录功能",
    "acceptance_criteria": ["支持邮箱登录", "OAuth 2.0", "JWT Token"],
    "deadline": "2026-06-05",
    "dependencies": ["T-001", "T-002"]
  },
  "timestamp": "2026-06-02T10:00:00Z",
  "context": { "project": "user-portal", "version": "2.0" }
}
```

#### 黑板模式（Blackboard Pattern）

> 所有 Agent 共享一个公共的"黑板"，在上面写入/读取信息，而不是直接点对点通信。

```
                  ┌────────────────────────────┐
                  │         📋 黑板             │
                  │  ┌───────────────────────┐ │
    Agent A ──────┼─>│ Task Queue             │ │
    (生产者)      │  │ [T1] [T2] [T3]         │ │
                  │  └───────────────────────┘ │
                  │  ┌───────────────────────┐ │
    Agent B ──────┼─>│ Results               │ │
    (消费者)      │  │ [T1: done] [T2: fail] │ │
                  │  └───────────────────────┘ │
                  │  ┌───────────────────────┐ │
    Agent C ──────┼─>│ Shared Context        │ │
    (观察者)      │  │ { project: "xyz" }    │ │
                  │  └───────────────────────┘ │
                  └────────────────────────────┘
```

**优点：** 解耦 Agent 之间的依赖关系，任意 Agent 只需关注黑板。
**缺点：** 黑板可能成为性能瓶颈。

### 2.3 任务分解与分配策略

#### 策略对比

| 策略 | 描述 | 适合场景 | Token 消耗 |
|------|------|----------|-----------|
| **自上而下分解** | Controller 一次性分解所有子任务 | 需求明确、步骤清晰的任务 | 中 |
| **动态分解** | 先做顶层分解，遇到问题时递归分解 | 复杂、不确定的任务 | 高 |
| **Greedy 分配** | 找最适合的 Agent 直接分配 | Agent 能力差异化大 | 低 |
| **Round-Robin** | 轮流分配 | Agent 能力同质 | 低 |
| **竞标制** | Agent 评估子任务后"投标"，Controller 选择 | 需要最优匹配 | 高 |

### 2.5 关键设计模式

#### 模式 1：Agent 交接（Handoff）

> 一个 Agent 发现自己不适合当前任务时，将任务转移给另一个 Agent。

```
Agent A (客服)
  → "用户需要重置密码"
  → Handoff to Agent B (安全) 
  → Agent B 处理重置
  → Handoff back to Agent A
  → "已为您重置，还需要其他帮助吗？"
```

**实现要点：**
- Handoff 函数返回一个信号（如 `{"handoff_to": "agent_b", "context": ...}`）
- 主循环检测到 Handoff 信号时切换 Agent
- 保持上下文连续性

#### 模式 2：监督者模式（Supervisor）

> 一个 Supervisor Agent 管理多个 Worker Agent，但不直接执行任务。

```
工作流程：
1. Supervisor 接收任务
2. Supervisor 分解为子任务
3. Supervisor 分配子任务给 Worker
4. 每个 Worker 独立执行，返回结果
5. Supervisor 收集结果、校验质量
6. 如果需要，指派 Worker 重做/改进
7. Supervisor 汇总输出
```

#### 模式 3：流水线模式（Pipeline）

> Agent 按顺序处理任务，前一个的输出是后一个的输入。

```
用户输入 → Agent A (分析) → Agent B (设计) → Agent C (编码) → Agent D (测试) → 输出

特点：
- 职责单一：每个 Agent 只做一件事
- 顺序依赖：前序 Agent 必须完成
- 易于理解和调试
- 适合有明确 Flow 的任务
```

#### 模式 4：投票/共识模式（Voting）

> 多个相同角色的 Agent 独立处理同一任务，然后投票或取平均得出最终结果。

```
            用户问题
         ↓ 广播给所有 Agent
   ┌── Agent A ──┐  ┌── Agent B ──┐  ┌── Agent C ──┐
   │  答案 A     │  │  答案 B     │  │  答案 C     │
   └─────┬───────┘  └─────┬───────┘  └─────┬───────┘
         └──────────┬─────┴───────┬──────────┘
                    ↓ 投票/取众数
                最终答案
```
