## Step 1 — 基础：为什么需要评估 Agent？

### 1.1 Agent vs 传统软件评估

```python
# 传统函数：可预测的输入→输出
def add(a, b):
    return a + b
assert add(1, 2) == 3  # ✅ 确定性测试

# Agent：不可预测的 LLM 驱动
def agent_add(a, b):
    response = llm(f"计算 {a}+{b} 等于多少？")
    return extract_number(response)
# 可能返回 3，也可能返回 "三个"、"three"、"3.0" 甚至算错！
```

**关键差异：**

| 维度 | 传统软件 | Agent |
|------|---------|-------|
| 输出 | 确定性 | 概率性 |
| 测试方法 | 断言精确值 | 需要语义评估 |
| 失败模式 | 明确（崩溃/报错） | 隐蔽（回答看似合理但错误） |
| 输入空间 | 有限组合 | 无限开放 |
| 复现难度 | 容易 | 困难（温度/随机性） |

### 1.2 Agent 评估的三层结构

```
┌──────────────────────────────────────┐
│         第 3 层：在线评估             │
│   A/B 测试 · 用户反馈 · 生产监控     │
├──────────────────────────────────────┤
│         第 2 层：系统评估             │
│   端到端任务 · 多轮对话 · 工具链      │
├──────────────────────────────────────┤
│         第 1 层：组件评估             │
│   LLM 调用 · RAG 检索 · 工具选择      │
└──────────────────────────────────────┘
```

- **组件层**：每个模块单独测试（RAG 检索准确率、Tool Call 精确率）
- **系统层**：完整的任务执行测试（用户说→Agent 做→结果检查）
- **在线层**：在生产环境中持续评估（用户满意度、任务成功率）

### 1.3 不评估的代价

| 场景 | 后果 |
|------|------|
| Agent 给出看似合理但错误的回答 | 用户误信造成损失 |
| Agent 调用了错误的工具 | 资金/API 消耗/数据泄露 |
| Agent 陷入死循环 | API 费用失控、响应超时 |
| Agent 产生有害输出 | 品牌损害、合规风险 |

> **核心认识：评估不是可选项，是 Agent 上线的必要条件。**

## Step 2 — 核心概念深入

### 2.1 Agent 评估指标体系

#### 🎯 任务级指标

| 指标 | 定义 | 计算方法 |
|------|------|----------|
| **任务成功率 (SR)** | Agent 成功完成任务的比率 | 成功数 / 总任务数 |
| **部分成功率 (PSR)** | 完成部分子任务的加权 | 权重 × 子任务完成度 |
| **平均步数** | 完成任务需要的 LLM 调用轮数 | 总步数 / 任务数 |
| **平均耗时** | 完成任务的时间 | 总时间 / 任务数 |
| **Token 消耗** | 完成任务消耗的 Token 数 | 输入+输出 Token 总和 |

#### 💡 推理质量指标

| 指标 | 定义 | 评估方法 |
|------|------|----------|
| **回答准确性** | 最终回答是否正确 | LLM-as-Judge / 人工标注 |
| **推理连贯性** | 每一步推理是否有逻辑 | 思维链一致性检查 |
| **召回率** | 关键信息是否被覆盖 | 预定义检查点 |
| **幻觉率** | 生成不存在信息的比例 | 事实核查 |

#### 🛠️ 工具使用指标

| 指标 | 定义 |
|------|------|
| **工具选择准确率** | 正确选择工具的比率 |
| **参数填充准确率** | 工具调用参数是否正确 |
| **工具调用次数** | 平均每个任务的工具调用数（过低可能少用，过高可能滥用）|
| **错误恢复率** | 工具失败后能否正确恢复 |

#### 🔒 鲁棒性指标

| 指标 | 说明 |
|------|------|
| **输入扰动容错** | 改变措辞后结果是否一致 |
| **边角情况处理** | 空输入、极长输入、模糊指令 |
| **攻击抵抗力** | Prompt 注入、越狱攻击 |
| **重复性** | 相同输入是否产生相似输出 |

### 2.2 评估方法论

#### 2.2.1 离线评估（Offline Evaluation）

**原理**：用标注好的测试集评估 Agent，不涉及真实用户。

**流程**：
```
1. 准备测试用例集（输入 + 期望输出 + 检查点）
2. Agent 逐一执行测试
3. 自动评估器打分
4. 生成评估报告
```

**数据集规模参考：**

| 阶段 | 用例数 | 目的 |
|------|--------|------|
| 开发期 | 20-50 | 快速迭代 |
| 测试期 | 100-500 | 覆盖率测试 |
| 发布前 | 1000+ | 全面评估 |

#### 2.2.2 在线评估（Online Evaluation）

**原理**：在生产环境中用真实用户评估。

| 方法 | 说明 |
|------|------|
| **A/B 测试** | 两版本同时运行，对比指标 |
| **用户反馈** | 点赞/踩、满意度评分 |
| **隐式信号** | 用户是否继续对话、是否复述问题 |
| **人工审核** | 抽样检查对话质量 |

#### 2.2.3 持续评估（Continuous Evaluation）

```
                     ┌─────────────┐
                     │  新数据入库  │
                     └──────┬──────┘
                            │
                     ┌──────▼──────┐
                     │  自动评估    │
                     │  Pipeline   │
                     └──────┬──────┘
                            │
              ┌─────────────┼─────────────┐
              │             │             │
        ┌─────▼─────┐ ┌────▼────┐ ┌────▼────┐
        │  通过      │ │  预警    │ │  失败   │
        │  正常部署  │ │  人工审查 │ │  回滚   │
        └───────────┘ └─────────┘ └─────────┘
```

#### 自建评估 Pipeline

**基本架构**：

```python
# 评估 Pipeline 骨架
class EvalPipeline:
    def __init__(self, agent, test_cases, evaluator):
        self.agent = agent
        self.test_cases = test_cases
        self.evaluator = evaluator

    def run(self):
        results = []
        for case in self.test_cases:
            prediction = self.agent.run(case.input)
            score = self.evaluator.evaluate(
                input=case.input,
                prediction=prediction,
                expected=case.expected
            )
            results.append(score)
        return self.summarize(results)
```

### 2.4 自动评估器设计

#### LLM-as-Judge

让 LLM 评估 LLM 的输出：

```python
evaluation_prompt = """
你是一个 AI Agent 评估专家。请评估以下 Agent 的输出：

[用户请求]: {user_input}
[Agent 输出]: {agent_output}

请从以下维度打分（1-5分）：
1. 准确性：回答是否正确？
2. 完整性：是否覆盖了用户的所有要求？
3. 工具使用：是否正确选择和使用了工具？
4. 效率：是否以最少的步骤完成任务？

请给出分数和理由。
"""
```

**注意事项**：
- 使用更强的模型做 Judge（如 GPT-4 评估 GPT-3.5）
- 避免位置偏见（交换评估顺序取平均）
- 需要校准（与人工评估结果对比）

#### 基于检查点的评估

对于复杂任务，预设关键检查点：

```python
checkpoints = [
    {"step": 1, "action": "调用搜索工具", "check": "search_tool in agent_steps"},
    {"step": 2, "action": "获取天气信息", "check": "weather in agent_output"},
    {"step": 3, "action": "给出出行建议", "check": "recommendation in agent_output"},
]
```

### 2.5 评估中的关键挑战

| 挑战 | 说明 | 解决方案 |
|------|------|----------|
| **评估成本** | 每次评估消耗 Token | 抽样评估、先小后大 |
| **标签稀缺** | 标注数据获取困难 | LLM 自动生成、主动学习 |
| **评估偏差** | LLM-as-Judge 有偏好 | 多重 Judge、校准 |
| **环境依赖** | Agent 调用外部 API | Mock / Sandbox |
| **非确定性** | 每次推理结果不同 | 多次运行取统计结果 |

## 🎯 面试题（8 道）

### Q1: Agent 评估和传统软件测试的本质区别是什么？

**答**：传统软件测试是**确定性**的——输入相同输出必定相同，测试用例可以精确断言。Agent 评估是**概率性**的——即使相同输入，LLM 也可能生成不同的输出。这意味着 Agent 测试需要：

1. 使用**语义匹配**而非精确匹配
2. 引入**统计指标**（均值、方差、置信区间）
3. 多次运行取统计结果
4. 增加**人工审核/LLM-as-Judge**环节

### Q2: 列举 5 个 Agent 评估的关键指标并说明用途。

| 指标 | 用途 | 计算方法 |
|------|------|----------|
| 任务成功率 | 衡量 Agent 完成任务的整体能力 | 成功数/总数 |
| 工具选择准确率 | 衡量 Agent 选择正确工具的能力 | 正确工具调用/总工具调用 |
| 平均步数 | 衡量 Agent 效率 | 总步数/任务数 |
| 幻觉率 | 衡量 Agent 事实准确性 | 幻觉语句/总语句 |
| 成本（Token 消耗） | 衡量经济可行性 | 输入+输出 Token 总和 |

### Q3: 什么是 LLM-as-Judge？有什么优缺点？

**答**：LLM-as-Judge 是用一个 LLM 模型来评估另一个 LLM 模型的输出质量。

**优点**：
- 减少人工标注成本
- 可规模化运行
- 能评估开放性回答的语义质量

**缺点**：
- 评估者有自身偏差（位置、风格、长度）
- 需要更强的模型做 Judge（成本更高）
- 可能产生"评估幻觉"

### Q4: 如何设计一个能够检测 Agent 死循环的评估？

**答**：可以从以下几个方面设计：

1. **步数上限**：设置最大 Step 数，超过即判失败
2. **模式检测**：检测重复的推理-行动-观察序列
3. **语义相似度**：检测连续输出的语义是否高度重复
4. **时间监控**：单步超时或总时间超限即终止

### Q5: 什么是评估 Pipeline？它一般包含哪些步骤？

**答**：评估 Pipeline 是将评估过程自动化的流水线，通常包含：

1. **数据加载**：加载测试用例集
2. **Agent 执行**：Agent 逐一执行任务
3. **自动评估**：用评估器对输出打分
4. **报告生成**：汇总结果、生成报告
5. **存档对比**：保存历史结果，支持对比

### Q6: 如何评估一个 RAG Agent 的检索质量？

**答**：从两个层面评估：

- **检索层面**：
  - Recall@K：关键文档是否在 Top-K 结果中
  - Precision@K：Top-K 结果中有多少相关文档
  - MRR：第一个相关文档的排名倒数均值
  - NDCG：带权重的排序质量

- **生成层面**：
  - 回答是否基于检索结果（归因检查）
  - 是否遗漏了关键信息（信息覆盖度）
  - 是否添加了检索结果之外的信息（幻觉检测）

### Q7: 在评估中如何处理 Agent 的非确定性（Non-Determinism）？

**答**：可以采用以下策略：

1. **多次运行**：每个测试用例运行 N 次（通常 3-5 次），取平均指标
2. **温度设置为 0**：虽然不保证完全确定，但可大幅降低方差
3. **种子重放**：一些 LLM 提供方支持 seed 参数
4. **置信区间**：报告指标时附带置信区间而非单一数值
5. **配对测试**：A/B 测试时使用相同的随机种子

### Q8: 从零开始建设一个 Agent 评估系统，你会怎么规划？

**答**：分四个阶段：

**阶段 1 — 手工评估（1-2 天）**
- 准备 20-50 个代表性测试用例
- 手动运行 Agent，人工评估
- 建立评估标准和流程

**阶段 2 — 自动化评估（3-5 天）**
- 实现基础评估 Pipeline
- 添加自动化检查点
- 引入 LLM-as-Judge
- 生成结构化报告

**阶段 3 — 系统化评估（1-2 周）**
- 扩展到 100+ 测试用例
- 实现多维度指标
- 建立回归测试
- 支持对比实验

**阶段 4 — 持续评估（持续）**
- 集成到 CI/CD Pipeline
- 在线监控和预警
- 用户反馈循环
- 自动更新测试数据集
